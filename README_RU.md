AIVPN не просто шифрует трафик — он делает его **неотличимым от обычной интернет-активности**. Для любого наблюдателя, включая государственные системы DPI, ваше соединение выглядит как обычный видеозвонок, веб-браузинг или любое другое повседневное приложение. Нет заголовков для фингерпринтинга. Нет хэндшейка для детектирования. Нет паттерна для сопоставления.

## Почему существующие VPN блокируются

У каждого популярного VPN-протокола — OpenVPN, WireGuard, Shadowsocks — есть отпечаток. Шифруйте сколько угодно, но *форма* трафика всё равно выдаёт: размеры пакетов слишком однородные, тайминги слишком ровные, хэндшейки следуют распознаваемой последовательности. Современные DPI-системы с ML-классификаторами ловят эти паттерны за секунды.

AIVPN решает это на уровне протокола:

| | Обычный VPN | AIVPN |
|---|---|---|
| **Подключение** | Детектируемый хэндшейк | Zero-RTT — данные идут с первого пакета |
| **Заголовки** | Известная структура / magic bytes | Динамические, уникальны для каждой сессии |
| **Форма трафика** | Однородная, машинная | Мимикрирует под реальные приложения |
| **Активные пробы** | Сервер отвечает ошибками | Полная тишина — невидим для сканеров |
| **Если утечёт исходный код** | Протокол скомпрометирован | Без последствий — безопасность в ключах, не в коде |
| **Если изъят сервер** | Прошлый трафик расшифровывается | Защищён — Perfect Forward Secrecy |

## Ключевые возможности

- **Адаптивная мимикрия трафика** — каждая сессия маскируется под профиль реального приложения
- **Zero-RTT старт** — нет хэндшейка, который можно детектировать; данные отправляются с первого пакета
- **Perfect Forward Secrecy** — эфемерный рэтчет ключей; компрометация сервера не раскрывает прошлые сессии
- **Невидимый сервер** — выглядит как закрытый порт для любого без валидных учётных данных
- **Криптографическая аутентификация сервера** — защита от MITM даже в полностью враждебной сети
- **O(1) маршрутизация пакетов** — определение сессии за константное время через ротирующиеся криптотеги
- **Безопасность памяти** — чистый Rust без `unsafe`; весь ключевой материал автоматически обнуляется при уничтожении
- **Минимальный размер** — клиент ~2.4 МБ, сервер ~2.6 МБ, работает на VPS за $5 с 1 ГБ RAM

### Что нового в 0.3.0

- **Нейронный резонанс (Baked Mask Encoder)** — для каждой маски запекается микро-нейросеть (MLP 64→128→64, ~66 КБ на маску), которая по реконструкции трафика определяет, детектирована ли маска DPI. Никаких внешних моделей — чистый Rust, работает на любом VPS.
- **Автоматическая ротация масок** — при обнаружении компрометации маски DPI-системой происходит немедленное переключение сессии на альтернативную маску из каталога без разрыва соединения.
- **Каталог масок с fallback-пулом** — регистрация, мониторинг статуса и автоматический выбор наилучшей доступной маски.
- **Детектор аномалий** — отслеживание packet loss и RTT по каждой маске; всплеск метрик указывает на DPI-блокировку.
- **Ротация ключей шифрования** — автоматическая ротация сессионных ключей по таймеру (120 с) и по объёму переданных данных (1 МБ).
- **Prometheus-метрики** — опциональный модуль мониторинга (feature flag `metrics`).

## Быстрый старт

### Требования

- **Rust 1.75+** — [rustup.rs](https://rustup.rs/)
- **Linux** с поддержкой TUN/TAP (сервер и клиент)
- **macOS** поддерживается для клиента и разработки
- **Root-доступ** для создания TUN-интерфейса и NAT

### Сборка

```bash
git clone <repo-url> && cd aivpn

# Сборка оптимизированных release-бинарников
cargo build --release

# Или через скрипт
./build.sh
```

### Запуск тестов

```bash
cargo test    # 113 тестов: крипто, протокол, мимикрия, нейронный резонанс, ротация масок, аномалии, ключи, стресс
```

## Настройка сервера

### 1. Генерация ключа сервера

```bash
sudo mkdir -p /etc/aivpn
openssl rand 32 | sudo tee /etc/aivpn/server.key > /dev/null
sudo chmod 600 /etc/aivpn/server.key
```

### 2. Запуск сервера

```bash
sudo ./target/release/aivpn-server \
    --listen 0.0.0.0:443 \
    --key-file /etc/aivpn/server.key
```

Сервер выводит свой **публичный ключ** при старте — передайте его клиентам.

| Флаг | По умолчанию | Описание |
|---|---|---|
| `--listen` | `0.0.0.0:443` | UDP адрес и порт прослушивания |
| `--key-file` | — | Путь к 32-байтному приватному ключу сервера |
| `--tun-name` | авто | Имя TUN-интерфейса (по умолчанию случайное) |
| `--config` | — | Путь к JSON-файлу конфигурации |

### 3. Включение NAT

```bash
sudo sysctl -w net.ipv4.ip_forward=1
sudo iptables -t nat -A POSTROUTING -s 10.0.0.0/24 -o eth0 -j MASQUERADE
```

### Конфиг сервера (опционально)

`/etc/aivpn/server.json`:
```json
{
  "listen_addr": "0.0.0.0:443",
  "tun_addr": "10.0.0.1",
  "tun_netmask": "255.255.255.0",
  "max_sessions": 500,
  "session_timeout_secs": 86400
}
```

## Настройка клиента

```bash
sudo ./target/release/aivpn-client \
    --server IP_СЕРВЕРА:443 \
    --server-key ПУБЛИЧНЫЙ_КЛЮЧ_BASE64
```

| Флаг | По умолчанию | Описание |
|---|---|---|
| `--server` | — | Адрес сервера (`ip:port`) |
| `--server-key` | — | Публичный ключ сервера (base64, 32 байта) |
| `--tun-name` | авто | Имя TUN-интерфейса |
| `--tun-addr` | `10.0.0.2` | IP-адрес туннеля клиента |
| `--config` | — | Путь к JSON-файлу конфигурации |

### Конфиг клиента (опционально)

```json
{
  "server_addr": "IP_СЕРВЕРА:443",
  "server_public_key": "КЛЮЧ_BASE64",
  "tun_addr": "10.0.0.2",
  "tun_netmask": "255.255.255.0"
}
```

## Как это работает

```
  Ваш трафик                                          Интернет
      |                                                   ^
      v                                                   |
+-----------+      неотличим от          +-----------+    |
|  AIVPN    | ==== обычного UDP =======> |  AIVPN    |----+
|  Клиент   |       (Zoom, HTTPS...)     |  Сервер   |
+-----------+                            +-----------+
```

1. Клиент перехватывает IP-пакеты через TUN-устройство, шифрует их и трансформирует трафик так, чтобы он был похож на конкретное реальное приложение
2. Каждый пакет несёт **ротирующийся криптотег**, который меняется с каждым пакетом и выглядит как случайные байты для любого наблюдателя
3. Сервер определяет сессии за O(1) через хэш-таблицу и полностью молчит для всего остального
4. При подключении автоматический **рэтчет ключей** обеспечивает forward secrecy — старые ключи уничтожаются, прошлый трафик остаётся защищённым

## Безопасность

| Свойство | Реализация |
|---|---|
| Шифрование | ChaCha20-Poly1305 AEAD, каждый пакет |
| Обмен ключами | X25519 с clamping и constant-time операциями |
| Forward Secrecy | Эфемерный рэтчет ключей, обнуление после использования |
| Аутентификация сервера | Верификация подписи Ed25519 |
| Защита от replay | 256-битный скользящий bitmap + монотонные счётчики |
| Гигиена ключей | `ZeroizeOnDrop` для всех секретов, `OsRng` для генерации |
| Ограничение нагрузки | Лимиты сессий по IP, троттлинг подключений |
| Нейронный резонанс | Baked Mask Encoder — ~66 КБ на маску вместо ~400 МБ LLM |
| Автоматическая ротация | Компрометация маски → мгновенное переключение на fallback |
| Ротация ключей | По таймеру и объёму данных, BLAKE3 key derivation |

Кодовая база прошла комплексный внутренний аудит безопасности. Все критические и высокоприоритетные уязвимости устранены.

## Структура проекта

```
aivpn/
├── aivpn-common/       # Крипто-библиотека, протокол, движок масок
├── aivpn-client/       # VPN-клиент
├── aivpn-server/       # VPN-сервер
│   ├── gateway.rs      #   UDP-шлюз, MaskCatalog, resonance loop
│   ├── neural.rs       #   Baked Mask Encoder, AnomalyDetector
│   ├── key_rotation.rs #   Ротация сессионных ключей
│   ├── metrics.rs      #   Prometheus-мониторинг (опционально)
│   └── passive_distribution.rs  # Пассивная раздача масок
├── config/             # Примеры конфигураций
├── Dockerfile          # Многоэтапная production-сборка
├── docker-compose.yml  # Контейнерное развёртывание
├── build.sh            # Скрипт сборки
└── setup.sh            # Настройка окружения для разработки
```
